# BinGO 体验与教材库下载改进设计

## 目标

本次改动覆盖网页版和 Tauri/Rust 桌面客户端，最终统一构建并发布 GitHub 版本。范围包括：

- 启动加载界面继承上次退出时的浅色/深色主题。
- 恢复 BinGO 原有 Logo，并统一启动页、导航、favicon、窗口和安装资源中的品牌图形。
- 重构作业模式的三栏布局，使主要任务、空状态和追问路径更清晰。
- 用户画像默认导出到 Windows 桌面，并支持选择文件夹及打开所在位置。
- 修复教材库“已在系统浏览器登录但 BinGO 下载失败”的问题，并提供明确的登录/网络错误反馈。

## 现状与根因假设

### 启动界面

`desktop-loader/loader.css` 当前写死了深色渐变和 `color-scheme: dark`。加载页未读取主应用保存的主题，因此会在浅色主题用户启动时仍显示深色。

### Logo

当前启动页只渲染文本 `BinGO`，窗口/系统资源和部分界面存在品牌资源不一致的风险。本次使用现有 BinGO 原始图标资源，不生成新的替代 Logo。

### 作业模式

当前页面为固定三栏空状态，左侧上传、中间内容、右侧追问同时占满空间。重构重点是任务优先级、空状态引导、响应式收缩，而不是改变作业求解 API 或数据模型。

### 用户画像导出

当前 `exportUserLearningProfileJson()` 只负责生成 Blob；调用层使用浏览器下载行为，成功 toast 没有显示路径，也没有打开文件夹或目录选择入口。

### 教材库

当前流程是：前端调用 `/api/textbooks/catalog`、`/search`、`/download`，本地 Node 服务端再通过 `proxyFetch` 访问国家中小学智慧教育平台的公开/资源接口。系统浏览器登录态不会自动进入 Node 服务端请求，因此浏览器中已登录不能保证 BinGO 请求已认证。截图中的 `fetch failed` 还隐藏了网络、上游状态码和认证失败的区别。

## 设计方案

### 1. 启动主题同步

启动页在首次绘制前读取与主应用一致的主题存储键。主题解析顺序为：

1. 明确保存的 `light` 或 `dark`。
2. 保存值为 `system` 时读取 `prefers-color-scheme`。
3. 无值时读取系统偏好。

加载页使用 Apple 设计系统的语义色：浅色背景为白色/浅灰，深色背景为 `#1C1C1E` 系列；移除紫蓝渐变和强光效。Logo 和加载卡片在两种主题下均保持足够对比度。

### 2. Logo 统一

建立单一品牌资源引用，优先复用仓库已有 BinGO 图标，而不是使用云同步占位图标。覆盖范围：

- desktop-loader 加载卡片。
- 应用 Header 和 favicon。
- Tauri 窗口图标、任务栏图标和 NSIS 图标。

### 3. 作业模式布局

采用“导入栏 + 主工作区 + 追问栏”的层级：

- 左侧导入栏固定窄宽，上传区是首要卡片，教材库和最近文件作为次级入口。
- 中间主工作区占据最大宽度，未导入时只展示一个明确的上传引导；有结果时展示题目、答案和解析。
- 右侧追问栏在未选题时收窄为轻量空状态，选中题目后展开；保留追问输入和上下文提示。
- 小窗口下按顺序收缩为主工作区优先，侧栏可折叠，不出现三个大面积空白面板。
- 颜色、间距、圆角、边框和阴影只使用语义 token，遵循 Apple 参考设计：低对比边框、克制阴影、清晰的系统蓝主操作色。

不改变作业求解、取消、流式状态和追问 API，只调整页面结构与展示状态。

### 4. 用户画像导出

默认行为：将 JSON 文件保存到 Windows 桌面。导出结果需要返回真实文件路径和文件名。

用户可选行为：

- “导出到桌面”：直接完成默认导出。
- “选择文件夹”：打开系统文件夹选择器，选择目标目录后导出。
- “打开所在文件夹”：调用 Windows 文件资源管理器定位已导出的文件。

导出成功 toast 显示文件名，并提供打开位置操作；失败提示包含可行动原因。

### 5. 教材库登录与下载

采用“BinGO 内置登录为主，现有 Windows 下载器为兜底”的方案。

#### 内置登录

- 教材库顶部显示登录状态。
- 未认证时提供“登录国家中小学智慧教育平台”入口，在 BinGO 内完成登录。
- BinGO 只保存完成教材接口请求所需的会话凭据，不读取 Edge/Chrome 的加密 Cookie。
- 目录、搜索、详情、PDF 下载统一由同一会话处理。
- 会话过期时显示“登录已过期，请重新登录”，不再只显示 `fetch failed`。

#### 下载错误分类

前端和 API 将区分：网络连接失败、上游 401/403、目录加载失败、资源不存在、PDF 流失败和文件超过大小限制。错误卡片提供重试、重新登录或打开下载器的对应操作。

#### Windows 下载器兜底

保留现有 `tchMaterial-parser-windows-x64.exe` 启动接口。内置接口不可用时，教材库提供“打开教材下载器”按钮，并在下载器完成后允许将 PDF 导入当前流程。

## 数据流

### 导出

`IndexedDB/localStorage snapshot -> desktop filesystem save dialog or Desktop default -> success result(path, filename) -> toast + Explorer reveal`

### 教材库

`TextbookLibraryDialog -> local auth/session -> /api/textbooks/catalog/search/download -> upstream platform -> normalized error or PDF stream -> File attachment`

系统浏览器登录态不作为隐式依赖；用户必须在 BinGO 内完成授权，或主动使用 Windows 下载器兜底。

## 验收标准

- 浅色主题退出后再次启动，加载页为浅色；深色主题同理。
- 启动页、窗口、导航和 favicon 使用同一 BinGO Logo，不出现云同步占位图标。
- 作业模式在大窗口和缩小窗口下均保持主任务优先，空状态不再撑满三栏。
- 用户画像默认出现在 Windows 桌面；可选择目录；可以打开资源管理器定位文件。
- 教材库未登录、登录过期、网络失败、权限失败和资源不存在时分别显示可理解的提示。
- 在 BinGO 内登录后可以加载目录、搜索教材并下载 PDF；失败时可重试或打开 Windows 下载器。
- 所有网页版变更通过 `pnpm desktop:build` 同步到 Tauri/Rust 客户端。
- 完成前端、服务端、相关 E2E 和桌面安装验证后，再生成签名资产、推送 GitHub 并发布版本。

## 发布策略

- 保留当前未提交的安装器模板变更，先纳入本次统一版本的发布提交。
- 版本号递增，构建时使用现有 updater 签名密钥。
- GitHub Release 上传安装器、更新器 zip、签名文件和 `latest.json`。
- 发布说明明确：主题同步、Logo 恢复、作业模式优化、画像导出、教材库登录/下载修复和旧版本升级注意事项。
