1. 执行摘要

BinGo 当前已经从“AI 生成课堂”升级为一个完整的教育操作系统 Runtime。系统以 Stage -> Scene -> Action 为课堂对象模型，以无状态 StatelessChatRequest 为大模型编排边界，以 SSE 结构化事件流驱动前端 Action Engine 和 Chat Session。

当前交付重点是三类能力已经闭环：第一，ICAP 的辩论与反向讲授机制可运行； 第二，白板视觉链路可被 Vision 模型读取；第三，生产级防御、测试、CI 和 telemetry 已接入；第四，Vision 低置信度场景会降级为澄清请求，不再让模型强行评价模糊白板。

3. ICAP Debate Flow 机制

3.1 辩论阶段模型

Debate Flow 使用 discussionMode: 'debate' 标记讨论 session。 后端 director 以 Agent A、Agent B、User 判决的固定阶段推进，避免普通讨论中发言权漂移。

  • 正方 Agent A 在白板左侧表达观点。
  • 反方 Agent B 在白板右侧表达观点。
  • 学生最终输入判决文本，系统将其作为批判性思维证据。

3.2 前端判决埋点

ChatAreaRef.getActiveDiscussionMode() 暴露当前 active discussion 的模式。 Stage.handleRoundtableMessageSend() 在识别 debate session 且用户输入非空时调用 trackEvent('icap_interaction', { type: 'debate_judgment' })。

4. Teach-back Vision 机制

4.1 阻塞与破局

wait_for_user_teaching 会在 Stage 中创建挂起 Promise。 学生需要完成白板编辑并发送讲解，系统才会 resolve。若学生点击 Need Hint，系统会发送隐藏消息 [USER_REQUESTED_HINT] 并退出学生主导状态。

4.2 多模态快照

Teach-back 提交时，Stage 调用 useCanvasStore.getState().getCanvasSnapshot() 获取白板 JPEG Base64，并通过 attachments 附加到用户消息。 use-chat-sessions.ts 将其转换为 OpenAI 兼容的 image_url 多模态 content。

快照导出只包含白板绘制结果，不包含工具栏等 UI 元素。导出逻辑加入了 JPEG quality 动态压缩，目标是将 Vision 请求图片控制在约 500KB 以内。

5. Vision Hallucination Fallback 机制

5.1 Prompt 级置信度约束

lib/orchestration/director-prompt.ts 新增 VISION_CONFIDENCE_FALLBACK_PROMPT。当最新用户消息包含白板图片时， Vision 模型必须先在内部评估“视觉清晰度与理解置信度（0-1）”。若 visionConfidence < 0.7，不得评价、纠错或推断学生意图。

5.2 降级 Action

lib/types/action.ts 新增 request_clarification 安全动作，并通过 SAFETY_ACTIONS 在 getEffectiveActions() 中强制可用。 这保证旧 Agent 配置未显式授权时，低置信度安全降级仍不会被动作过滤器丢弃。

5.3 编排短路与可见反馈

lib/orchestration/director-graph.ts 在解析到 request_clarification 后，会优先合成 AI 气泡文本： “你画的这部分线条有点密集，我看得不是很确切，你能用语音再补充解释一下这里的连线逻辑吗？” 随后发出 action 事件并停止处理该 Agent 后续输出，避免模型在低置信度下继续生成幻觉式评价。

6. 白板账本、摘要与 Token 防御

白板编辑事件会被记录为 WhiteboardActionRecord，并随 directorState.whiteboardLedger 注入下一轮无状态请求。为防止上下文膨胀， 当前实现设置 MAX_LEDGER_ENTRIES = 10。

5.1 智能摘要预留

当账本超过 10 条时，系统将早期记录压缩为一条 { type: 'summary', content, sourceCount }，后续保留最近 9 条原始记录。 代码处保留 // TODO: call summarization AI，当前用轻量本地摘要函数占位。

5.2 全量编辑器事件桥接

WhiteboardEditorTrace 统一承接未来 Fabric.js / Konva 等底层引擎事件。 已监听并规范化 object:modified、path:created、 selection:deleted，再写入 recordStudentWhiteboardAction()。

8. 学习画像闭环

lib/learning/profile-engine.ts 已接入 Teach-back 与 Debate 两类 ICAP 证据。

  • 学生点击 Need Hint 时生成“建构受阻/需引导”证据，响应文本为 USER_REQUESTED_HINT。
  • Debate 结束后抽取学生最终判决，生成“批判性思维/观点判决”证据。
  • 证据统一写入学习画像更新流，供跨课时生成和个性化补救使用。

9. 生产级基建与可观测性

8.1 CI/CD

.github/workflows/ci.yml 已配置 main 分支 push/PR 门禁。 工作流安装 pnpm 依赖后依次执行：

  • pnpm exec eslint .
  • pnpm exec tsc --noEmit
  • pnpm exec playwright install chromium --with-deps
  • pnpm exec playwright test

8.2 LLM Telemetry

app/api/chat/route.ts 已记录每次请求的 latency、估算 input/output token、 session 标签、debate 标签和 vision-teach-back 标签。 lib/telemetry.ts 保留 // TODO: send to LangSmith/Helicone。

前端 trackEvent() 已预留 PostHog/Mixpanel 接口。没有接入真实 SDK 时， 开发环境会输出调试日志，生产环境不会阻塞课堂流程。

10. 性能优化与测试覆盖

9.1 Dynamic Import

白板组件已从静态引入改为 next/dynamic 懒加载。 components/canvas/canvas-area.tsx 只在 whiteboardOpen 为 true 时加载白板 chunk，并提供稳定尺寸的 loading skeleton，减少首屏 FCP 压力和布局抖动。

9.2 E2E Vision 测试

tests/e2e/teach-back-vision.spec.ts 模拟 Teach-back 模式下用户作图并发送讲解。 测试通过 page.route('/api/chat') 拦截请求，断言 messages 中同时包含文本与 格式正确的 Base64 image_url。
